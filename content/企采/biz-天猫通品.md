# 天猫通品业务逻辑

## 三期迭代历史

| 期次 | 时间 | 核心内容 |
|------|------|----------|
| 一期 | 2025-04 | 商家报名、类目品牌资质申请、供应商管理、天猫数据审核 |
| 二期 | 2025-07 | 天猫订单回流、账单时间设定、售后地址、自动开票 |
| 三期 | 2025-09 | 品牌修改、价格巡检 + 破价自动下架 |

## 审批链路（一期）

| 步骤 | 说明 |
|------|------|
| 商家提报超链审核 | 审核通过后，渠道商品管理页面可见 |
| 商家补全天猫信息 | 再次进入审核流程 |
| 全渠道铺货审核通过 | **两个必须条件**：企采已为"供货中" + 已补全天猫商品信息 |
| 审核顺序 | 先审核超链，才能审核全渠道铺货 |
| 审核通过后 | 商家侧**无退出全渠道的入口** |

> 商品提报提交后，仍可点击全渠道铺货（此时两个审批同时进入源选审批管理）

## 订单链路

```
渠道订单（天猫侧）→ 履约单（供应商在 work 后台/万采操作发货）→ 结算单
```

逆向链路：
- 买家在天猫申请退款 → 小二在天猫平台操作同意/拒绝
- 退款结果**自动更新**到渠道订单和履约单（供应商不介入）
- **支持部分退**

## 天猫渠道定价规则

- **天猫渠道默认包邮**：买家下单不额外收运费，运营定价时将运费计入渠道售价
- 渠道售价由运营手动设定，须覆盖含税含运费供货价及利润空间

**供货价拆解公式：**
```
含税含运费供货价 = 含税供货价 + 平均运费
purchase_price（含运）= supply_price（含税）+ (purchase_price - supply_price)（运费）
```

| 业务概念 | 对应 DB 字段 | 所在表 |
|---------|------------|--------|
| 渠道商品 SKU 售价 | `channel_price` | `qycg_mall_channel_offer_item` |
| 含税供货价 | `supply_price` | `qycg_mall_supply_product_item` |
| 含税含运费供货价 | `purchase_price` | `qycg_mall_channel_sale_order_detail` |
| 买家侧运费（包邮） | `out_post_fee = 0` | `qycg_mall_channel_sale_order_detail` |

> 注意：`qycg_mall_product.min_supply_price_with_post_fee` 测试品可能为 NULL，以 `order_detail.purchase_price` 替代。

详细数据链路与 SQL 见 [[tech-天猫价格链路]]。

## 价格巡检（三期重点测试）

- **破价处理**：价格巡检发现破价 → 发 MQ 消息 → 自动下架商品
- **并发风险**：消息重试机制可能并发触发多次下架，需通过锁机制（乐观锁/DB 唯一索引）保证幂等

## 测试验证要点

1. `channel_offer_item.channel_price` = 订单子单 `sale_price`（分）
2. 公式：`purchase_price = supply_price + (purchase_price - supply_price)`
3. 包邮验证：`out_post_fee = 0`
